### 📜 Conventional Commits (규약 커밋) 의 기본 구조

가장 기본적인 구조는 다음과 같습니다.

`<type>(<scope>): <subject>`

- **`<type>` (필수):** 커밋의 종류를 나타냅니다. (가장 궁금해하신 부분)
- **`<scope>` (선택):** 커밋이 영향을 미치는 범위를 나타냅니다. (예: `api`, `login`, `ui`)
- **`<subject>` (필수):** 커밋에 대한 짧은 요약 설명 (보통 50자 이내, 명령문으로 작성)
- **`<body>` (선택):** 변경 사항에 대한 자세한 설명.
- **`<footer>` (선택):** 이슈 트래커 ID (예: `Closes: #123`)나 주요 변경점(Breaking Change)을 명시.

---

### ✨ 주요 커밋 유형 (Type) 정리

가장 자주 사용되는 핵심 유형들입니다.

| **Type** | **설명** |
| --- | --- |
| **`feat`** | **새로운 기능** 추가 (feature) |
| **`fix`** | **버그 수정** (bug fix) |
| **`docs`** | **문서** 수정 (documentation) |
| **`style`** | 코드 스타일 수정 (세미콜론 추가, 공백 제거 등. **코드 로직 변경 없음**) |
| **`refactor`** | 코드 **리팩토링** (기능 변경 없이 코드 구조 개선) |
| **`test`** | **테스트 코드** 추가 또는 수정 |
| **`chore`** | 빌드 작업, 패키지 매니저 설정 등 **기타 잡일** (chore) |

### 🛠️ 기타 유용한 유형들

프로젝트 규모가 커지거나 세분화할 때 유용한 유형들입니다.

| **Type** | **설명** |
| --- | --- |
| `build` | 빌드 시스템 또는 외부 종속성 변경 (예: npm, webpack, gradle) |
| `ci` | CI/CD 설정 파일 변경 (예: Github Actions, Jenkins) |
| `perf` | 성능 개선 (performance) |
| `revert` | 이전 커밋으로 되돌리기 (revert) |

### Examples 실제 적용 예시

**예시 1: 간단한 버그 수정**

> fix(login): 소셜 로그인 시 리다이렉트 URL 오류 수정
> 

**예시 2: 새로운 기능 추가 (자세한 내용 포함)**

> feat(api): 사용자 프로필 이미지 업로드 API 추가
> 
> - 사용자가 자신의 프로필 이미지를 업로드할 수 있는 엔드포인트(`POST /api/v1/users/profile-image`)를 구현했습니다.
> - Amazon S3를 스토리지로 사용합니다.
> 
> `Closes: #45`
> 

**예시 3: 코드 스타일 수정**

> style(core): 전체 파일에 대해 Prettier 포맷팅 적용
> 

**예시 4: chore 작업**

> chore: .gitignore 파일에 build 로그 파일 추가
> 

이 규칙을 사용하면 Git 로그가 훨씬 더 깔끔하고 전문적으로 관리됩니다.
